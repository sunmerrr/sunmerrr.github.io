### 게임 세팅 전체 UI 및 로직 변경
**내용**: 게임 생성이 조금 더 적은 뎁스를 거쳐서 될 수 있도록 하기위해 리뉴얼 함 + 하는김에 리팩토링까지

1. 코드 흐름 개선: useEffect가 남발되어서 코드 흐름이 제대로 읽히지 않던 부분 리팩토링
    

#### 게임 생성 개선
- **히스토리**    
  기존에 토너먼트 게임의 경우는 게임이 `생성-(수정)-배포요청-배포` 순으로 작업되어 있었다.    

  구현은 대충 이렇게 되었었다.    
  설정에 꼭 필요한 것들만 첫번째 뎁스에 노출     
  -> 두번째 뎁스를 반드시 지나나야하며 해당 부분에서 게임 배포 가능     
  -> 두번째 뎁스에서 다른 상세 게임을 편집할 수 있는 2-1, 2-2, ...5 까지 접근 가능
  -> 이런식으로 2, 3 뎁스 안에 게임을 만듬(`생성-(수정)-배포`)     
  -> Web3가 포함되면서 바로 배포될 수 없고 `배포요청 및 대기` 단계를 추가하게 됨      
  -> 괴물과 같은 모습으로 변함  
 
#### 문제점

- 역할 분담이 제대로 이뤄지지 않음     
처음에는 form 과 useContext 만 같이 있었다.    
이후에 useContext 로 인한 너무 잦은 전체 리랜더링 때문에 zustand 를 넣었다. 복잡한 유효성 검사를 쉽게 하기 위해서 zod를 도입했다.      

#### 결과
- zustand 를 도입하면서 제대로 걷어내지 못한 useContext    
  한동안(최근까지) useContext 와 zustand 가 계속해서 같이 쓰이고 있었다. 게임 상태 저장을 위해서 초기에 useContext 를 사용한 부분이 어디서부터 어디까지 걷어내야할지 감이 안잡혔다는 변명을 해볼 수 있겠다.    
  아무튼 이번에 v3를 작업하면서 죄다 걷어냈다.(후련)    
  사실상 useContext는 거의 무용지물이었고, 그냥 레거시여서 쉽게 걷어낼 수 있었다.
- 유효성 검사를 위한 zod 도입, 하지만 커스텀 훅에서 실행되는 유효성 검사(?)     
  유효성 검사를 편하게 하기 위한 도구로 도입하게 됐는데 이것 또한 제대로 관리가 되지 않았다.     
  예를 들자면 title 글자 수 유효성 검사는 zod 에 있으면서도 특수문자 검사는 커스텀 훅에 있는 식으로 스파게티가 만들어졌다.     
  그냥 title 하나정도만 있으면 수정하기 수월했겠지만, 우리가 만들고있는 설정 페이지는 생각보다 까다로웠다.        
  하지만 너무 복잡하거나 zod 에서는 하지 못하는 로직도 있었어서 어쩔 수 없는 로직 한 두개를 빼고는 모두 옮겨주었다. 코드가 훨씬 깔끔해졌고, 유지보수하기도 수월해졌다.
- 제대로 된 역할 분담
  위의 내용과 이어지지만 여기에 React-Hook-Form 이 더해진다
  값 A 바뀌면 상태 배열 B,C,D 3개를 업데이트 해줘야 한다던가. B 배열을 바꾸면 조건에 따라서 A,C 만 업데이트 되어야 한다와 같은 조건이 여러개가 있었고, 서로 연관되어있었다.이런 부분에서 버그도 많았고, 예상하지 못한 패턴을 발견하게 되어서 아주 애를 먹었었다.     

  기획자와 논의 하면서 v3 에서는 서로의 의존성을 많이 끊어줄 수 있었고, 조금 더 깔끔하게 동작하는 코드를 만들게 되었다.

### 통계&분석 서비스 작업
#### 신규 서비스 작업
- 없는 데이터..
- 라이브러리에서 지원하지 않는 ui

#### 문제점
- 알지만 모르는 도메인    
  내가 몸담고 있는 곳의 서비스는 게임쪽이다. 내가 여기서 하고 있는일은 게임 생성, 인벤토리 확인, 내 게임 히스토리 확인 같은 것이다.    
  여기에 분석 서비스를 넣어주어야 하는데, 게임을 설정하는 것에는 익숙하지만 게임을 어떻게 분석할지에 대해서는 많이 무지했다. 그래서 무엇을 어떻게 보여주는지 잘 알지 못하는 상황이었고, 해당 도메인의 서비스들이 보통 비싼 유료 컨텐츠라서 레퍼런스를 보는 것도 좀 어려웠다.    
  이런 연유로 인해서 처음 기획서에 나와있는 약식의 ui 를 보고서 조금 당황스러웠다. 어떤 데이터를, 왜, 어떻게, 무엇을 얻고자 보는 것인지 시각적인 요소를 보면서도 이해하지 못했기 때문이었다.    
  다행히 기획해주신 분이 그 비싼 유료 서비스를 사용해보시고, 이해도도 있으신 분이었어서 도움을 많이 받을 수 있었다.    
- 디자이너의 부제    
  기존에 디자인 해주시던 웹 디자이너 분이 퇴사하시고, 다음분이 오실때까지의 공백 기간이 있었다. 나는 딱 그 공백기에 해당 프로젝트를 진행하게 되었다. 기존에 계시던분이 잡아뒀던 몇개의 페이지를 조합하고, 제대로 나오지 않았떤 아이콘은 대충 내가 만들고(ㅜ) 
- 서버의 부제(? 조금 늦게 개발 되었을 뿐)    
  일단 데이터를 어떻게 보여줄지, 타입을 어떻게 할지 같은것을 이야기했으면 좋았겠지만.. 당시에 서버가 너무나 바빠서 일단 프론트에서 먼저 그려보기로 했었다.     
  그런데 나도 잘 모르는 데이터와 잘 모르는 UI를 보여주려니 목데이터를 만드는 것도 쉽지 않았다.     
- 촉박한 일정    
  아직 데이터도 안나왔는데 일단 와꾸부터 보고 싶다던 저 위에 사람들이 있어서 후다닥 작업해야했다.     
  하지만 라이브러리 선택부터 조금 어려웠다.

### 부사수 채용
사수도 있어본 적 없는 나의 짧은 생에 처음으로 부사수가 입사했다.    
어떻게 해줘야 좋은 사수일지 고민해봤지만 뜻대로 되지도 않았고 고민했던 것보다 훨씬 어려웠다. 그리고 나와 같은 마음이 아닐때가 많아서 허들이 많았던 것 같다.    
여전히 어떻게 하는것이 좋은 사수인지는 모르겠다. 그냥 내가 받았던 좋았던 것들을 천천히 다 해주면 좋지 않을까 하는 막연함만 있는 것 같다.     
     
덕분에 2026년에는 조금 더 도전적인 한 해가 될 것 같다는 생각도 들고, 많이 성장 할 수 있을 것 같은 기대도 있다.
